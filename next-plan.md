## 1. 作るもの（推奨設計）

**「調整＆採点エクスプローラ」** — 1枚の静的 Web（同じ GitHub Pages に追加、サーバ不要）。

体験の流れ:
1. ゲームを選ぶ（タブ）。
2. スライダーで「打ち方」を動かす（例 Hanabi＝捨て札比・色ヒント比・数字ヒント比…）。
   または「実プレイヤーをランダム読込」でスライダーを実データの1人にセット。
3. その場で更新：
   - **採点ゲージ**：あなたに近い実プレイヤーは技量◯パーセンタイル。
   - **助言チップ**：あなたが上位層に届いていない項目だけ「▲◯◯を増やす／▼△△を減らす」。
   - **近い実プレイヤー例**：実際の技量ラベル（Elo/花火点/スコア…）を併記＝データ由来の裏付け。

### 中核メカニズム＝実プレイヤーへの近傍探索（kNN）
- ブラウザ内で、入力した打ち方に**最も近い実在プレイヤー k 人（k≈25）**を標準化特徴空間で探す。
- 採点＝近傍の z₁ パーセンタイル平均（＋実技量も表示）。
- **モデル近似なし・100%実データ由来**（このプロジェクトの正直さ原則に合致）。サーバ/推論エンジン不要。
- 助言＝各レバーで「上位1/3の平均」に届かない向きを表示（phase2.py と同じ tercile 方向）。

## 3. 実装ステップ

1. **データ書き出し（Python, 新規 `gamesnf/export_app.py`）**：対象ゲームごとに JSON を作る。
   - 実プレイヤー配列：各行 `{feat_std:[…], feat_raw:[…], z1_pct, skill, skill_pct}`。
     大きいゲーム（チェス7712/Hanabi1858）は ~800 に間引く。
   - 特徴メタ：`{name, idx, min, max, step, is_lever}`（NAMES/LEVERS は phase2.py に既定義）。
   - 標準化 `mu/sd`、上位1/3平均（標準化）＝助言の到達目標、tercile improve 方向。
   - z₁ は `phase2._z1_all`（キャッシュ `$SCRATCH/games/<key>/z1_all.npy`）を再利用。
   - 出力先 `docs/app_data/<key>.json`。
2. **ブラウザアプリ（`docs/play.html`, 自己完結・インライン JS/CSS）**：
   - タブ＝ゲーム、各特徴スライダー（初期＝中央値）、「実プレイヤー読込」ボタン。
   - JS：標準化→kNN(k=25)→採点ゲージ＋助言チップ＋近い実プレイヤー例。
   - 既存ページのカラー/フォント（Hiragino Sans）を踏襲。日本語の「観察/解釈」注記を添える。
3. **導線**：`docs/index.html` の Phase 2 節に「▶ 触って体験する」リンク（→ play.html）。
   逆に play.html から index.html へ戻るパンくず。
4. **公開**：commit→push（Pages自動更新）。`python -m gamesnf.export_app` で JSON 再生成。

## 4. 技術メモ・ハマりどころ（前セッションの学び）

- z₁ は per-game 教師つき軸を全ユニットに適用（`_z1_all`）。チェスは 2500 サブサンプル学習＋
  npy キャッシュ（別プロジェクト `merlnf.train` が CPU を占有すると 15倍遅くなる→他人のプロセスは
  殺さずサブサンプル＋キャッシュで回避）。
- 助言は**必ず tercile（上位1/3 vs 下位1/3）平均差の向き**で。1例 vs 近傍だと z₁ が甘いと逆転し
  お手本が弱くなる／Hanabi で既知傾向と逆の助言が出る（前回踏んだ罠）。
- お手本の「一歩上」検索は **z₁ ではなく実技量**で（z₁ が甘いと帯が逆転する）。
- 図/文字：Hiragino Sans、`<b>` 使用（Markdown ** はHTMLで描画されない）、日本語文字列に ASCII " を入れない。
- ページ再生成は `python -m gamesnf.games page`（results.json + crossgame.json + phase2.json から、再学習なし）。
